# ПКФ СНАРК — Исполнительная геодезическая документация

Веб-приложение на **Streamlit** для автоматизации исполнительных листов **вертикальности опор контактной сети трамвая**. Один исполнительный лист соответствует **одной опоре и одному PDF** (при необходимости дополнительно генерируется DXF по шаблону).

Проект реализован **модульно** (не монолитный файл): парсинг проекта и замеров, расчёт отклонений по **ГОСТ Р 51872-2024**, экспорт в **PDF** (основной выход), **Excel** (сводка) и опционально **DXF**.

## Возможности

- **Проект (Вариант А)**  
  - Загрузка **PDF проекта** (обязательно) — ведомости, типы опор, при необходимости координаты из текста/таблиц.  
  - Опционально **DXF проекта** — точные плановые координаты и высоты по слоям (`0_Point_Symbols`, `0_Point_Name`, `0_Point_Height` и др.).  
  - Если DXF не загружен, координаты опор берутся только из PDF (менее надёжно для плановых координат).

- **Геодезические замеры**  
  - Файлы **TXT**, **DXF**, **XML** (до 10 файлов за сессию обработки в интерфейсе).  
  - Привязка точек к опорам по **имени точки** и по **расстоянию в плане** (порог настраивается в `config.py`).

- **Расчёт**  
  - ΔX, ΔY в миллиметрах, полное отклонение √(ΔX² + ΔY²), угол направления отклонения.  
  - Сравнение с **допуском** по ГОСТ Р 51872-2024 (относительный L/150 и абсолютные пороги по типам опор).

- **Выход**  
  - **PDF** на каждую опору: мини-схема с красной линией отклонения, таблица проект/факт, штамп ПКФ СНАРК, логотип (если есть в `assets/`).  
  - **Сводный Excel** по всем опорам.  
  - **ZIP** со всеми PDF и Excel.  
  - Опционально **DXF** по пользовательскому шаблону.

- **Интерфейс**  
  - Обзор проекта с **картой Plotly**, редактируемая таблица опор.  
  - Сохранение проектов в `data/projects/`, экран **«Загруженные проекты»**.  
  - Предпросмотр PDF одной опоры, прогресс-бары, настройки штампа в боковой панели.

## Конфигурация

### Переменные окружения

| Переменная           | Описание |
|----------------------|----------|
| `OPENROUTER_API_KEY` | Ключ API OpenRouter (для AI-парсинга сложных страниц PDF). Если пусто, используются только pdfplumber и эвристики. |

Ключ можно также ввести в **боковой панели** приложения (сессия; для продакшена лучше переменная окружения или секреты хостинга).

### Файл `config.py`

- **`OpenRouterConfig`**: базовый URL, модели (`gemini-2.5-pro` для парсинга PDF, `deepseek-r1` зарезервирована под расширения), лимиты токенов.  
- **`GOSTTolerances`**: делитель **150** для относительного допуска, минимум/максимум мм, словарь абсолютных допусков по типам опор (`ТФГ-1500-10`, `ТФГ-2300-10`, `DEFAULT` и т.д.).  
- **`AppConfig`**: `match_radius_m` — радиус привязки точек замеров к опоре (по умолчанию **2 м**).  
- **`StampConfig`**: организация, стадия, подписи (частично редактируются в UI).  
- **`DXFLayers`**: имена слоёв для парсера проекта и генератора DXF.

### Каталоги

- `data/projects/` — сохранённые проекты: `project.pdf`, `project.dxf` (если был), `metadata.json`, подпапка `measurements/` для загруженных замеров.  
- `templates/` — сюда пользователь может положить постоянный **DXF-шаблон** или загружать его через UI при генерации.  
- `assets/logo_snark.png` — логотип в шапке PDF и в боковой панели (при наличии файла).

## Пользовательский сценарий (UI)

### Боковая панель

- Логотип (если есть `assets/logo_snark.png`).  
- Навигация: **Новый проект**, **Загруженные проекты**, **Обзор проекта**, **Замеры**, **Генерация**.  
- **Настройки штампа**: геодезист, проверил, ГИП.  
- **OpenRouter API Key** (опционально).

### 1. Новый проект

1. Ввести **название проекта** (обязательно).  
2. Загрузить **PDF проекта**.  
3. При необходимости загрузить **DXF проекта**; если DXF нет — показывается предупреждение.  
4. Нажать **«Обработать проект»** — файлы копируются в `data/projects/<имя>/`, выполняется парсинг.  
5. При отсутствии автоматически распознанных опор показывается предупреждение: опоры можно **добавить вручную** на экране обзора.

### 2. Загруженные проекты

Список каталогов с `metadata.json`: краткая информация и кнопка **«Открыть»** — повторная загрузка данных из сохранённых PDF/DXF.

### 3. Обзор проекта

- Метрики: число опор, число уникальных типов, признак загрузки DXF.  
- **Карта Plotly**: рассеяние по X/Y (при ненулевых координатах).  
- **Редактор таблицы опор** (`st.data_editor`): добавление/правка номера, типа, высоты, координат.  
- Переход к экрану замеров.

### 4. Замеры

- Многофайловая загрузка TXT/DXF/XML.  
- Кнопка **«Обработать замеры»** — прогресс, таблица по файлам (имя, число точек, распознанные опоры в именах точек).  
- Сводная таблица отклонений с подсветкой статуса.  
- Переход к **генерации**.

### 5. Генерация

- Выбор опор (мультиселект).  
- Опциональная загрузка **DXF-шаблона** для генерации DXF.  
- Флаги: генерировать **PDF**, **DXF**.  
- **«Создать исполнительные листы»** — прогресс по опорам, формирование ZIP (PDF + Excel + при необходимости DXF).  
- **«Предпросмотр»** — генерация PDF для первой выбранной опоры (встраивание через base64; в некоторых браузерах поведение iframe может отличаться).  
- Повторное скачивание ZIP, если архив уже был сформирован в сессии.

## Входные данные

### PDF проекта

- Используется **pdfplumber**: поиск страниц с ключевыми словами (ведомость, опоры и т.п.), попытка извлечь таблицы.  
- При наличии **API-ключа OpenRouter** — страница рендерится в изображение и отправляется в **Gemini 2.5 Pro** с промптом на JSON-массив опор (номер, тип, высота, координаты при наличии).  
- **Fallback**: эвристики по тексту (regex по номерам опор и типовым обозначениям).

**Важно:** экспорт PDF из САПР часто даёт «плохие» таблицы для автоматического извлечения; в сложных случаях опоры проще **дополнить или ввести вручную** на экране обзора.

### DXF проекта (типичный Civil/план)

- Центры опор: **CIRCLE** на слоях вроде `0_Point_Symbols` / `Опоры проект`.  
- Подписи: **TEXT** на `0_Point_Name` (например `N317` → нормализуется до `317`).  
- Высоты: **TEXT** на `0_Point_Height`.  
- Сопоставление текста к центру круга по расстоянию в плане (порог порядка единиц чертежа в метрах).

### TXT замеров (пример формата)

Строки без заголовка. Поддерживаются два порядка колонок (переключатель в боковой панели на экране замеров):

**Вариант 1 — X, Y, Z** (по умолчанию):

```text
573A.3,74204.183,119389.735,22.087
```

**Вариант 2 — Y, X, Z** (частая выгрузка: сначала «северная», затем «восточная» координата):

```text
574.3,74156.863,119387.010,21.578
```

- Имя точки, затем две плановые координаты и **Z** через запятую (десятичный разделитель — точка или запятая в числах).  
- Имена вида **`573A.1`**, **`574.2`**: префикс — номер опоры, суффикс после точки — номер замера.  
- Точки стояния вида **`1 (34)`** помечаются как станции и **не участвуют** в привязке к опорам.

### DXF замеров

- **Приоритетный формат** (типично выгрузка с полевого ПО): вставки блока **`Measured`**, координаты — точка вставки **INSERT**; номер точки берётся с ближайшей подписи на слое **`Nomer`** (радиус поиска ~0,15 м, настраивается в `config.py`: `dxf_measurement_*`).
- Если таких вставок нет — используется запасной режим: **POINT** и ближайший **TEXT** в радиусе ~3 м.

**Пояса (3+3):** если на одной опоре оказывается больше шести точек, для расчёта оставляются **три с минимальным Z** и **три с максимальным Z** (промежуточные съёмки отбрасываются). Пороги задаются в `config.py`: `pole_belt_lower_count`, `pole_belt_upper_count` (по умолчанию 3 и 3).

### XML

- Поддержаны распространённые варианты разметки (в т.ч. элементы с координатами в дочерних тегах); при нестандартном формате может потребоваться доработка парсера под ваш прибор.

## Алгоритмы и нормативка

### Вертикальность (упрощённо)

1. Для каждой опоры точки замеров делятся на **нижнее** и **верхнее** сечение:  
   - по суффиксам (например `.1` — нижнее, `.2`/`.3` — верхнее при трёх точках на опору);  
   - либо по половине набора точек по **Z**, если суффиксов нет.  
2. В плане считаются **центры** нижней и верхней группы точек.  
3. **ΔX, ΔY** — разность координат центров (в метрах), в отчёте — в **миллиметрах**.  
4. **Отклонение** = √(ΔX² + ΔY²) в мм.  
5. **Угол** — азимут направления отклонения в градусах (от оси Y, по часовой стрелке), см. `utils/geometry.angle_from_north`.

Если для опоры недостаточно точек для разделения на два уровня, но есть проектные координаты и несколько фактических точек, используется **запасной режим**: проектный центр как «низ», среднее фактических точек как «верх» (см. `deviation_calculator._calculate_from_project_center`).

### ГОСТ Р 51872-2024

В коде реализована **практическая модель допуска**:

- относительная часть: **H/150** (H — высота опоры в мм);  
- сравнение с **абсолютным** порогом по типу опоры из словаря в `GOSTTolerances`;  
- итоговый допуск ограничен **min/max** в мм.

Статусы:

- **Норма** — отклонение не превышает ~80% допуска;  
- **Предупреждение** — до 100% допуска;  
- **Превышение** — выше допуска.

Точные формулировки норматива для вашего объекта следует сверять с **полным текстом ГОСТ Р 51872-2024** и проектной документацией; при необходимости таблицы в `config.py` и логику в `gost_checker.py` можно уточнить под конкретный договор.

## Выходные документы

### PDF (основной)

- Формат **A3 альбомная** (через `reportlab`, страница `landscape(A3)`).  
- Блоки: рамка, заголовок, **мини-схема** отклонения (красная линия, стрелка), **вид сбоку** (схематично), таблица параметров, штамп с реквизитами, ссылка на ГОСТ.  
- Шрифты: попытка **DejaVu** (кириллица); при отсутствии системных TTF — **Helvetica** (латиница; кириллица в PDF может отображаться некорректно — на Windows обычно ставят полные шрифты или используют встраивание TTF в доработках).

### Excel

- Лист со сводной таблицей, заголовком проекта, датой, цветовой подсветкой столбца «Статус».

### ZIP

- Имена PDF: `ИС_<номер_опоры>.pdf`, файл `Сводная_таблица.xlsx`, при включении DXF — `ИС_<номер>.dxf`.

### DXF (опционально)

- Копирование шаблона с заменой части текстов и добавлением линии отклонения; сложные шаблоны Civil могут потребовать **ручной донастройки** слоёв и плейсхолдеров под ваш файл.

## OpenRouter и модели

- Базовый URL: `https://openrouter.ai/api/v1` (см. `OpenRouterConfig`).  
- Для распознавания таблиц с PDF-страницы по умолчанию: **`google/gemini-2.5-pro-preview-03-25`** (идентификатор модели может меняться у провайдера — при ошибках API проверьте актуальный список моделей на сайте OpenRouter).  
- В `config` также указан **`deepseek/deepseek-r1`** для возможных будущих сценариев (аналитика, не обязателен для текущего UI).

Запросы идут через **httpx**; заголовки включают `Authorization: Bearer <ключ>`.

## Ограничения и рекомендации

1. **Качество PDF проекта** — не всегда восстанавливаются таблицы; используйте DXF для координат и ручную правку таблицы опор.  
2. **Два уровня съёмки** — для корректной вертикальности нужны точки, однозначно разделяемые на нижний и верхний пояс; при одном поясе из трёх точек расчёт опирается на эвристики или запасной режим.  
3. **Кириллица в PDF** — зависит от наличия TTF в системе; для продакшена имеет смысл явно подключить шрифт с кириллицей в `pdf_exporter.py`.  
4. **Безопасность** — не коммитьте API-ключи; используйте переменные окружения и `.gitignore` для `data/projects/` при необходимости.  
5. **DXF шаблон** — структура чертежей из Civil 3D уникальна; автозаливка всех полей без настройки под конкретный шаблон может быть неполной.

## Устранение неполадок

| Проблема | Что проверить |
|----------|----------------|
| Пустой список опор после PDF | Указать DXF; вручную добавить опоры; задать `OPENROUTER_API_KEY` для AI-парсинга |
| Нет отклонений после замеров | Совпадение номеров опор в проекте и в именах точек; радиус `match_radius_m`; наличие координат у опор |
| Ошибка кодировки в консоли Windows | `PYTHONIOENCODING=utf-8` |
| Streamlit не запускается | Виртуальное окружение, `pip install -r requirements.txt`, свободный порт 8501 |
| PDF без кириллицы | Установка шрифтов DejaVu или правка `pdf_exporter` для встраивания TTF |
